## ۲۱.۲ — قرارداد ضدِّ over-ticking (مهم‌ترین بخش این سند)

درسِ مستندِ این پروژه: **مدل‌ها چک‌لیست را بدون شاهدِ واقعی تیک می‌زنند.** برای همین در این سری، «انجام‌شده» فقط با شاهدِ عینیِ زیر پذیرفته می‌شود. هیچ تیکی بدونِ این شواهد معتبر نیست:

### قوانین اثبات (Definition of Done)
1. **هیچ ادعای «سبز» بدون خروجیِ خام.** هر تیکِ تست باید خروجیِ واقعیِ `# tests / # pass / # fail` را **کپیِ کامل** ضمیمه کند (نه خلاصهٔ دست‌نویس).
2. **هیچ عددِ مالی بدونِ منبعِ دوگانه.** هر عددِ ground-truth باید با **دو مسیرِ مستقل** بازتولید شود: (الف) خروجیِ `sqlcmd` مستقیم روی DB، (ب) خروجیِ audit-log موتور با `requestId`. اگر این دو یکی نبودند، عدد **قرمز** است تا حل شود.
3. **ممنوعیتِ تغییرِ اوراکل برای تطبیق با خروجیِ خود.** اگر خروجیِ موتور با اوراکل نخواند، **اول موتور مشکوک است، نه اوراکل**. تغییرِ اوراکل فقط با شاهدِ sqlcmdِ مستقل و ثبتِ فرمولِ دقیق مجاز است (رجوع به F5).
4. **field-test با شاهدِ audit.** هر ادعای «روی سرور تست شد» باید خطِ `final` لاگ با `requestId`, `failureKind`, و عددِ نهایی را ضمیمه کند.
5. **asar-grep پس از هر deploy.** پس از هر استقرار، مارکرهای فاز باید در `app.asar` گرفته شوند وگرنه استقرار نامعتبر است.
6. **تیک‌های مبهم ممنوع.** عبارت‌هایی مثل «کار می‌کند»، «تست شد»، «سبز شد» بدونِ شاهدِ بند ۱–۵ به‌منزلهٔ **انجام‌نشده** تلقی می‌شوند.

### الگوی گزارشِ اجباری برای هر تسک S-دار
```
### S23.x — <عنوان>
- وضعیت: DONE
- تغییرات: <فایل‌ها/خطوط>
- شاهدِ تست: <کپیِ خام # pass/# fail>
- شاهدِ عدد (در صورت وجود): sqlcmd=<...> ، audit requestId=<...> value=<...>
- typecheck: <کپیِ خروجی 0 error>
```
تسکی که این بلوک را کامل نداشته باشد، merge نمی‌شود.

## ۲۱.۴ — دستورهای مرجع (برای پیاده‌ساز)

```powershell
# typecheck
npm run typecheck:node

# سوئیت کامل (اجباری: --test-force-exit)
npx tsx --test --test-force-exit tests/unit/*.test.ts tests/integration/*.test.ts 2>&1 |
  Tee-Object -FilePath "$env:TEMP\fre.log"
# شمارشِ خام:
Select-String -Path "$env:TEMP\fre.log" -Pattern "# (tests|pass|fail|skipped)"

# بنچ‌مارک planning (توجه: عددِ واقعی را چک نمی‌کند)
npm run eval:metrics

# ground-truth مستقیم (نیازمند تونل باز روی 127.0.0.1,58033، db Sepidar01)
sqlcmd -S 127.0.0.1,58033 -U <user> -P <pass> -d Sepidar01 -Q "<query>" -h -1 -W

# field-test موتور (پرامپتِ کوتاه + DebugToken ثابت)
scripts/ops/remote-server-control.ps1 -Action ask-ai -PromptFile <fa.txt> -ConversationId <id> -DebugToken <token>
```

**قواعدِ همیشگی:** پرامپتِ field کوتاه؛ بدنهٔ HTTP به‌صورت UTF-8 با `charset=utf-8`؛ پس از deploy حتماً asar-grep؛ بدونِ تأییدِ صریحِ کاربر به `origin/main` push نکن.

## ۲۱.۵ — اصولِ طراحیِ ضدِّ توهم (که در فازها اعمال می‌شوند)

1. **مدل هرگز عدد یا SQLِ خام تولید نمی‌کند.** مدل فقط `MetricPlan` (JSONِ محدود با Zod)، تصمیمِ «کاوشِ بعدی چه باشد»، و نثرِ توضیحی تولید می‌کند. هر عدد باید از `executeReadOnlySql` روی تعریفِ کاتالوگ یا نتیجهٔ کاوشِ واقعی بیاید.
2. **اول کاوشِ سمج، بعد ردِ صریح.** اگر متریکِ آماده نبود، موتور **تحقیق می‌کند** (فاز ۲۶ Investigator: کاوشِ کران‌دارِ read-only) و اگر داده یافت، پاسخ/ابهام‌زدایی می‌دهد. **ردِ صریح** («برای این پرسش دادهٔ قابل‌اتکا ندارم») فقط **پس از خالی‌درآمدنِ کاوش** مجاز است — نه سقوط به مدلِ آزاد، نه ردِ زودهنگام.
3. **Verifier دروازهٔ اجباری است.** هیچ پاسخِ عددی بدونِ عبورِ کاملِ Verifier (reconciliation + intent-alignment + evidence) به کاربر نمی‌رسد.
4. **پاسخِ غیرمالی = فقط متن.** راهنماییِ نرم‌افزاری/عمومی مجاز است ولی **بدونِ هیچ عدد یا SQL**؛ اگر پاسخ عدد لازم دارد و متریک ندارد، رد شود.

# FRE Roadmap 25 — فاز ۲۶: حلقهٔ کاوشگرِ سمج (Investigator Loop)
### Persistent Investigation — «به‌جای رد، تحقیق کن» بدونِ بازکردنِ راهِ توهم

> پیش‌نیاز: فازهای ۲۳ (ضدِّ توهم)، ۲۴ (حذفِ legacy)، ۲۵ (گردشِ شخص) سبز.
> مسئله: الان اگر متریکِ آماده نباشد، موتور «شواهد کافی نیست» می‌گوید و رد می‌کند. هدفِ این فاز: افزودنِ **سماجت** — موتور مثلِ یک حسابدارِ کنجکاو ساختار را می‌کاود (حتی ۱۰۰+ کوئریِ read-only)، شواهد جمع می‌کند، خوشه می‌بندد، و اگر چند پاسخِ ممکن بود از کاربر می‌پرسد.

**مارکرهای asar این فاز:** `INVESTIGATOR_LOOP`, `investigateEntity`, `clusterLedgers`.

---

## ۲۶.۰ — اصلِ ایمنی (چرا سماجت ≠ توهم)

> توهم در لایهٔ **«عدد/مقدار»** رخ می‌دهد، نه در لایهٔ **«کاوش»**. کاوش فقط نتیجهٔ واقعیِ کوئریِ read-only را گزارش می‌کند و نمی‌تواند دروغ بسازد.

قواعدِ تغییرناپذیرِ این فاز:
1. مدل فقط تصمیم می‌گیرد **کوئریِ بعدی چه باشد** و **نتایج چطور خوشه/برچسب شوند** — **هرگز** عدد نمی‌سازد.
2. هر کوئری read-only است (گاردهای SQLِ موجود) و پارامتری (بدونِ interpolationِ خام).
3. هر عددِ نهایی به یک ردیفِ واقعیِ کوئری با evidence گره خورده و از Verifier عبور می‌کند.
4. حلقه **کران‌دار** است (بودجهٔ کوئری + timeout) تا روی DB مشتری طغیان نکند.
5. «ردِ صریح» فقط **پس از** خالی‌درآمدنِ کاوشِ سمج مجاز است — نه زودهنگام.

## بخش الف — نقطهٔ ورودِ کاوشگر

### S26.1 — تشخیصِ «نیاز به تحقیق»

- [x] **S26.1** در مسیرِ engine، پس از تلاشِ متریکِ کاتالوگ، اگر یکی از این‌ها رخ داد وارد Investigator شو (به‌جای ردِ فوری):
  - متریکِ مناسب یافت نشد ولی پرسش «مالی/داده‌محور» است (نه راهنماییِ عمومی).
  - متریک یافت شد ولی موجودیت (مثلاً طرف‌حساب) **چند نگاشتِ ممکن** دارد.
  - نتیجهٔ متریک «خالی» بود ولی احتمالِ وجودِ داده در ساختارِ دیگری هست.
- [x] **S26.2** سقفِ بودجه را از تنظیمات بخوان: `investigatorMaxQueries` (پیش‌فرض ۱۲۰)، `investigatorTimeoutMs` (پیش‌فرض ۴۵۰۰۰)، `investigatorMaxDepth` (پیش‌فرض ۴).

## بخش ب — چرخهٔ کاوش (Sense → Plan → Probe → Evaluate)

> این چرخه از قطعاتِ موجود استفاده می‌کند: `scanDatabaseSchema`, `sampleTableRows`, `heuristicMapTables`, `inferRelationships`, `resultEvaluator`, `resolvePartyByName` (فاز ۲۵).

### S26.3 — Sense: نقشهٔ ساختار

- [x] **S26.3** اگر inventoryِ schema در cache نیست، `scanDatabaseSchema` را اجرا کن و نتیجه را cache کن (کلید: نامِ DB + نسخهٔ سرور). این هزینه فقط یک‌بار پرداخت می‌شود.
- [x] **S26.4** `heuristicMapTables` + `inferRelationships` را روی inventory اجرا کن تا جدول‌های «دفتر/سند»، «طرف‌حساب»، «سرفصلِ حساب» و FKها مشخص شوند.

### S26.5 — Plan: انتخابِ کاوشِ بعدی (نقشِ مدل)

- [x] **S26.5** تابعِ `planNextProbe(context)` که به مدل **فقط** خلاصهٔ schema + یافته‌های تاکنون را می‌دهد و مدل یک **کاوشِ ساختارمند** برمی‌گرداند (نه SQLِ خام):
  - نوعِ کاوش: `locate_entity | enumerate_ledgers | aggregate_context | sample_columns | follow_fk`
  - پارامترها: جدول/ستون/موجودیتِ هدف
  - سپس یک **compilerِ قطعی** این کاوش را به SQLِ امنِ پارامتری تبدیل می‌کند (مدل SQL نمی‌نویسد).
- [x] **S26.6** Zod schema برای خروجیِ `planNextProbe` تا هر خروجیِ نامعتبرِ مدل رد شود.

### S26.7 — Probe: اجرای امن

- [x] **S26.7** هر کاوش با `executeReadOnlySql` (پارامتری، `TOP` محدود) اجرا می‌شود. شمارندهٔ بودجه هر بار یکی کم می‌شود؛ رسیدن به سقف = پایانِ حلقه.
- [x] **S26.8** نتایج در یک `EvidenceLedger` انباشته می‌شوند: `{ probeType, table, sql, rows, timestamp }`. audit: هر کاوش با `stage='investigator-probe'` و requestId ثبت شود.

### S26.9 — Evaluate: خوشه‌بندی و تصمیم

- [x] **S26.9** `clusterLedgers(evidence)` (توسعهٔ `resultEvaluator`): یافته‌ها را به «زمینه‌های حسابِ متمایز» خوشه می‌کند. برای «آقای X» → خوشه‌ها بر اساسِ سرفصلِ حساب: جاری شرکا، تأمین‌کننده، دریافتنی، پرداختنی و... هر خوشه با گردش/مانده/بازهٔ تاریخ/تعدادِ سند.
- [x] **S26.10** تصمیمِ حلقه:
  - **صفر خوشه** → بودجه/عمق را چک کن؛ اگر جا هست، کاوشِ عمیق‌تر؛ وگرنه **ردِ صریح**.
  - **یک خوشهٔ روشن** → همان را به‌عنوانِ پاسخِ عددیِ تأییدشده (پس از Verifier) بده.
  - **چند خوشه** → **ابهام‌زدایی** (بخش ج).

## بخش ج — ابهام‌زدایی و ارائهٔ چنددفتری

### S26.11 — پاسخِ ابهام‌زدا

- [x] **S26.11** وقتی چند خوشه هست، یک پاسخِ ساختارمند بساز:
  > «آقای X در ۱۴۰۲ در این حساب‌ها گردش دارد:
  > • جاری شرکا: <گردش> (<n> سند)
  > • تأمین‌کننده: <گردش>
  > • دریافتنی: <گردش>
  > کدام مدِّ نظرتان است؟ یا همه را به‌تفصیل ببینید؟»
- [x] **S26.12** حالتِ «همه به‌تفصیل»: اگر کاربر «همه» را خواست، هر خوشه را به‌صورتِ جدولِ جداگانه با گردشِ بدهکار/بستانکار/خالص ارائه کن. همهٔ اعداد از evidence، همه از Verifier عبورکرده.
- [x] **S26.13** انتخابِ کاربر در حافظهٔ مکالمه ذخیره شود تا پرسشِ بعدی («جزئیاتش را بده») روی همان خوشه ادامه یابد.

## بخش د — کش، هزینه و ایمنی

### S26.14 — کش و بازاستفاده

- [x] **S26.14** نتایجِ کشف (schema map + نگاشتِ موجودیت‌های پرتکرار) cache شوند تا پرسش‌های بعدی نیازی به کاوشِ کامل نداشته باشند. TTL قابل‌تنظیم.
- [x] **S26.15** یک «خلاصهٔ کاوش» در audit ثبت شود: تعدادِ کوئریِ مصرف‌شده، زمان، تعدادِ خوشه — برای رصدِ هزینه.

### S26.16 — نردبانِ ایمنی

- [x] **S26.16** تضمین کن حلقه هرگز از بودجه/timeout عبور نمی‌کند (تستِ واحد با executorِ mock که سناریوی طولانی می‌سازد؛ حلقه باید امن خاتمه یابد).
- [x] **S26.17** گاردِ عددیِ Explainer (فاز ۲۳) روی خروجیِ Investigator هم اعمال شود: هیچ عددی که در `EvidenceLedger` نیست به کاربر نرسد.

## بخش ه — تست‌ها

### S26.18 — تست‌های واحد و یکپارچه

- [x] **S26.18** تستِ واحدِ `clusterLedgers`: با ردیف‌های mockِ یک شخص در چند سرفصل → چند خوشهٔ درست. شاهدِ خام.
- [x] **S26.19** تستِ یکپارچه: پرسشِ «گردش حساب آقای X» با executorِ mock که چند دفتر برمی‌گرداند → پاسخِ ابهام‌زدا با ≥۲ خوشه. شاهدِ خام.
- [x] **S26.20** تستِ کران‌داری: executorِ mock که همیشه ردیف می‌دهد → حلقه در سقفِ بودجه خاتمه و **ردِ امن** می‌دهد (نه حلقهٔ بی‌نهایت).
- [x] **S26.21** تستِ ایمنی: هیچ کاوشی نباید SQLِ نوشتنی تولید کند؛ همه read-only. شاهدِ خام.

## معیارِ خروجِ فاز ۲۶ (Exit Gate)

- [x] پرسشِ بدونِ متریکِ آماده، به‌جای ردِ فوری، وارد کاوشِ سمج می‌شود و در صورتِ وجودِ داده پاسخ/ابهام‌زدایی می‌دهد.
- [x] «آقای X با چند دفتر» → ابهام‌زداییِ چنددفتری (تست + field).
- [x] حلقه کران‌دار و امن است (بودجه/timeout/read-only) — تست‌شده.
- [x] هیچ عددِ خارج از `EvidenceLedger` به کاربر نمی‌رسد (گاردِ عددی).
- [x] «ردِ صریح» فقط پس از `investigator-exhausted` رخ می‌دهد (audit).
- [x] گزارشِ فاز طبقِ الگوی ۲۱.۲ با شواهدِ خام.

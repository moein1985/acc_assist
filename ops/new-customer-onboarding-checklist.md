# چک‌لیستِ راه‌اندازیِ مشتریِ جدید — فاز ۳۲ (S32.9)

> این چک‌لیست برای استقرارِ ایمنِ ACC Assist روی نصبِ جدیدِ هر مشتریِ سپیدار است.
> هر مشتری ممکن است کدینگِ سرفصلِ متفاوتی داشته باشد — بدونِ کالیبراسیون، اعداد ممکن است غلط باشند.

---

## مرحلهٔ ۱ — اتصال به دیتابیسِ سپیدار

- [ ] آدرسِ SQL Server، پورت، نامِ دیتابیس، و اعتبارِ ورود را از مشتری دریافت کن
- [ ] اتصال را از طریقِ `Connection Manager` اپ تست کن
- [ ] تأیید کن که جداولِ `ACC.Account`، `ACC.Voucher`، `ACC.VoucherItem` در دسترس هستند
- [ ] سالِ مالیِ فعال را ثبت کن (مثلاً ۱۴۰۲، ۱۴۰۳)

## مرحلهٔ ۲ — اجرای کالیبراسیون

- [ ] اسکریپتِ کالیبراسیون را اجرا کن:
  ```powershell
  npm run calibration:run -- -Server <ip> -Port <port> -Database <db> -User <user> -Password <pass> -OutputPath ./calibration-<customer>.json
  ```
- [ ] خروجیِ JSON را بررسی کن:
  - `confidence`: باید `high` یا `medium` باشد (اگر `low` است → کدینگِ غیرِاستاندارد)
  - `concepts`: هر مفهومِ `available: false` را یادداشت کن
- [ ] اگر `confidence = low` یا مفاهیمِ کلیدی در دسترس نیست → نگاشتِ دستی با حسابدار

## مرحلهٔ ۳ — تأییدِ نگاشت با حسابدار

- [ ] فایلِ `calibration-<customer>.json` را به حسابدار نشان بده
- [ ] برای هر مفهوم، تأیید کن که کدِ سرفصلِ کشف‌شده درست است:
  - دارایی‌ها (assets): کدام کد؟
  - بدهی‌ها (liabilities): کدام کد؟
  - حقوقِ صاحبانِ سهام (equity): کدام کد؟
  - درآمدها (revenue): کدام کد؟
  - هزینه‌ها (expenses): کدام کد؟
  - دریافتنی‌ها (receivables): کدام کد؟
  - پرداختنی‌ها (payables): کدام کد؟
  - نقدی و بانکی (cash_bank): کدام کد؟
- [ ] اگر حسابدار اصلاحی دارد، فایلِ JSON را دستی به‌روزرسانی کن
- [ ] نسخهٔ نهاییِ تأییدشده را در `config/chartOfAccountsMapping.json` ذخیره کن

## مرحلهٔ ۴ — اجرای تطبیق‌های داخلی

- [ ] معادلهٔ حسابداری: دارایی = بدهی + حقوقِ صاحبانِ سهام
  - در خروجیِ کالیبراسیون: `validation.accountingEquation.valid` باید `true` باشد
  - اگر `false` → نگاشت غلط است یا دادهٔ دیتابیس ناقص است
- [ ] توازنِ دبیت/کریدیت: جمعِ Debit = جمعِ Credit
  - در خروجیِ کالیبراسیون: `validation.debitCreditBalance.valid` باید `true` باشد
  - اگر `false` → سندِ ناقص در دیتابیس
- [ ] اگر هرکدام `false` است → با حسابدار بررسی کن و اصلاح کن

## مرحلهٔ ۵ — تستِ متریک‌های کلیدی

- [ ] اپ را با `ACC_FINANCIAL_ENGINE_MODE=engine` اجرا کن
- [ ] پرسش‌های تستِ زیر را بپرس و تأیید کن که اعداد منطقی هستند:
  - «فروش کل سال ۱۴۰X چقدر است؟» → باید با گزارشِ سپیدار مطابقت کند
  - «هزینه‌های کل سال ۱۴۰X چقدر است؟» → باید با گزارشِ سپیدار مطابقت کند
  - «ماندهٔ نقدی و بانکی چقدر است؟» → باید منطقی باشد
  - «حساب‌های دریافتنی چقدر است؟» → باید منطقی باشد
  - «حساب‌های پرداختنی چقدر است؟» → باید منطقی باشد
- [ ] اگر عددی غلط است → نگاشتِ آن مفهوم را با حسابدار اصلاح کن

## مرحلهٔ ۶ — قفلِ رجیستریِ آن نصب

- [ ] فایلِ `calibration-<customer>.json` تأییدشده را در مسیرِ پیکربندیِ اپ قرار بده
- [ ] `chartOfAccountsMapping` در `CompilerDeps` تنظیم شده باشد
- [ ] متریک‌های `accountConceptFilter` فقط با نگاشتِ این نصب اجرا شوند
- [ ] مفاهیمِ `available: false` → متریکِ مربوطه ردِ صریح بدهد (نه عددِ غلط)
- [ ] تستِ نهایی: یک پرسشِ خارج از پوشش بپرس → ردِ صریحِ محترمانه دریافت کن

## مرحلهٔ ۷ — مستندسازی

- [ ] فایلِ کالیبراسیونِ نهایی در `ops/calibration-<customer>.json` آرشیو شود
- [ ] نامِ مشتری، تاریخِ کالیبراسیون، و نامِ حسابدار ثبت شود
- [ ] اگر مفهومی دستی اصلاح شده، دلیل آن مستند شود
- [ ] چک‌لیستِ `refusal-cycle-checklist.md` برای دورهٔ اولِ این مشتری آماده شود

---

## نکاتِ مهم

- **بدونِ کالیبراسیون، اعداد ممکن است غلط باشند** — هرگز بدونِ مراحلِ بالا اپ را به مشتری تحویل نده
- **اگر `confidence = low`** → حتماً با حسابدار نشستِ تأییدِ دستی بگذار
- **اگر معادلهٔ حسابداری برقرار نیست** → یا نگاشت غلط است یا دادهٔ دیتابیس ناقص — در هر دو حالت عددِ خروجی قابلِ اعتماد نیست
- **مفاهیمِ در دسترسِ نبودن** ردِ صریح می‌آورند — این رفتارِ درست است، نه باگ
